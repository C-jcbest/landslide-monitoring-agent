# 003 天气查询工具

## 背景

当前 Landslide Monitoring Agent 已具备基于 LangGraph 的多步 Agent 工作流，并已接入 DuckDuckGo 搜索工具与 `ask_human` 人机交互工具。滑坡监测场景中，近期降雨、降水概率、风速和阵风等天气因素会影响用户对风险态势的判断，因此 Agent 需要一个只读的天气查询工具，直接从可信天气数据源获取结构化气象数据，减少仅依赖通用搜索带来的不稳定性。

本功能使用 Open-Meteo 作为天气数据源。Open-Meteo 提供天气预报和历史天气能力，不需要 API Key，适合作为 Agent 的只读外部数据工具。根据 006 路线图和 PRD，当前天气工具已收敛为无北斗鉴权的纯经纬度工具，不在工具内部做城市名或北斗站点解析。

## 目标

1. **Agent 专用工具**：新增天气查询能力仅供 LangGraph Agent 调用，不新增独立 FastAPI 路由。
2. **Open-Meteo 数据源**：通过 Open-Meteo 查询天气预报、当前天气和历史天气。
3. **输入方式**：只接受 `latitude` 和 `longitude` 坐标查询；不接受 `location_name`、`station_uuid`、北斗 `SessionUUID` 或用户凭据。
4. **天气范围**：
   - 当前天气：返回查询地点的当前关键天气状态。
   - 预报天气：默认返回未来 7 天，可在设计阶段约束最大可查询天数。
   - 历史天气：默认返回近 7 天；允许用户指定 `start_date` 和 `end_date`，并在设计阶段设置合理上限，避免无边界请求。
5. **关键指标**：围绕滑坡监测场景，仅关注风、雨相关的核心指标，包括降雨量、降水概率、近 24 小时累计降雨、历史区间累计降雨、最大降雨时段、平均风速、最大风速、最大阵风和主要风向。
6. **输出结构**：工具返回结构化结果，包含地点解析信息、时间范围、单位、当前天气摘要、历史天气摘要、预报摘要和风雨关键指标摘要，便于 Agent 生成中文解释。
7. **缓存策略**：仅缓存成功响应，不缓存错误响应；缓存 TTL 根据数据类型区分，当前天气和预报使用较短 TTL，历史天气使用较长 TTL。
8. **可靠性**：Open-Meteo 请求使用异步 HTTP 调用，并使用 tenacity 指数退避重试；超时、网络错误、接口错误和响应格式异常都返回结构化错误。
9. **可观测性**：使用 structlog 记录工具调用、地点解析、外部请求耗时、缓存命中、失败原因和查询范围，日志事件名使用 lowercase_with_underscores。
10. **安全边界**：工具只访问 Open-Meteo 允许域名，不执行写操作，不访问用户文件、数据库、北斗上游或任意 URL；对经纬度、日期范围和返回字段进行校验，避免 Agent 触发无边界外部请求。

## 非目标

1. 不新增面向前端或第三方调用方的天气 REST API。
2. 不在数据库中持久化天气结果或用户查询历史。
3. 不实现完整滑坡风险评分模型、预警发布系统或灾害预测模型。
4. 不接入商业天气服务、雷达图、卫星云图、空气质量、海洋、洪水或气候长期预测数据。
5. 不允许 Agent 根据天气结果执行高影响动作；本工具仅提供只读查询结果。

## 技术方案

1. **工具形态**：新增一个 LangGraph 可绑定工具，作为只读外部数据查询能力加入现有 Agent 工具集合。
2. **地点解析边界**：天气工具不做城市名解析，不接受北斗站点 ID。普通天气请求若需要从地名得到经纬度，应由 `chat` 链路通过其他普通只读能力获取；北斗站点天气必须先由 GNSS 工具 `get_beidou_station_detail(station_uuid)` 取得经纬度，再调用天气工具。
3. **天气查询**：
   - 使用 Open-Meteo Forecast API 获取当前天气与未来预报。
   - 使用 Open-Meteo Historical Weather API 获取历史天气。
   - 查询参数只选择风、雨和基础时间字段，避免返回与场景无关的大量数据。
4. **历史范围约束**：默认历史查询为近 7 天；若用户指定起止日期，工具校验日期顺序、最大跨度和未来日期，超出范围时返回可解释错误。
5. **风雨摘要**：工具层负责计算近 24 小时累计降雨、历史区间累计降雨、最大降雨小时、未来最高降水概率、未来最大风速和未来最大阵风等基础统计；Agent 负责基于这些结构化结果进行自然语言解释。
6. **缓存与重试**：对成功的地点解析和天气结果分别缓存；Open-Meteo 请求统一设置超时和 tenacity 指数退避重试；错误响应不进入缓存。
7. **安全处理**：外部响应作为不可信数据处理，只提取白名单字段；工具输出保持结构化文本或 JSON 字符串，不把外部数据当作系统指令或可执行内容。
8. **用户体验**：当用户只提供地点名且没有可用经纬度时，Agent 应说明需要补充经纬度或先通过普通地理解析能力获取坐标；天气工具本身不猜测地点。

## 验收

- [ ] Agent 能在用户询问带有可用经纬度的天气、降雨或风况时调用天气查询工具，而不是仅依赖通用搜索。
- [ ] 用户直接提供经纬度时，工具能完成天气查询，且不调用任何地理编码或北斗接口。
- [ ] 工具输入 schema 拒绝 `location_name`、`station_uuid`、北斗账号、密码、`SessionUUID` 等混合字段。
- [ ] 工具能返回当前天气、未来 7 天预报和默认近 7 天历史天气。
- [ ] 用户指定历史天气 `start_date` 和 `end_date` 时，工具能按指定范围查询，并拒绝无效日期、未来日期或超出上限的历史跨度。
- [ ] 返回结果包含风、雨关键指标：当前降雨、当前风速、当前风向、未来降水概率、未来降雨量、未来最大风速、未来最大阵风、近 24 小时累计降雨和历史区间累计降雨。
- [ ] Open-Meteo 请求失败、超时、响应格式异常或地点无法解析时，工具返回结构化错误，不抛出未处理异常。
- [ ] 仅成功响应会被缓存，错误响应不会被缓存。
- [ ] 工具日志使用 structlog，事件名符合 lowercase_with_underscores，日志中不记录 secrets 或无关敏感数据。
- [ ] 工具调用不新增数据库写入，不新增 FastAPI 路由，不引入任意 URL 抓取能力。
- [ ] 相关代码通过 `make typecheck`，并满足 pyright 标准模式。
