# 004 北斗凭据绑定与校验

## 背景

项目后续需要让智能体基于当前用户绑定的北斗监测平台账号获取监测点、GNSS 实时数据和日监测数据，并在对话中完成分析。根据 `docs/reference/beidou-platform-api.md` 的约束，运行时不配置全局北斗业务账号，普通用户查询、Agent 工具、邮件订阅和定时分析都必须使用当前用户或订阅所属用户绑定的北斗凭据。

当前项目已有本地账号、会话认证和 LangGraph Agent 基础能力，但尚未建立用户级北斗凭据绑定、保存前验证、加密存储、会话缓存和授权缺失提示机制。若不先完成本功能，后续 5 节点图结构中的权限门禁和 GNSS 数据执行节点无法可靠落地。

## 目标

1. **用户级凭据绑定**：每个需要访问北斗/GNSS 上游数据的本地用户都必须单独绑定北斗平台账号密码，不支持全局北斗账号兜底。
2. **绑定状态查看**：用户可以查看自己的北斗凭据绑定状态；已绑定时返回绑定的北斗用户名。
3. **凭据录入与更新**：普通用户可以自行录入或更新自己的北斗平台账号密码。
4. **保存前验证**：保存凭据前必须调用北斗登录认证接口验证上游账号密码可用；验证失败不得保存新凭据。
5. **加密存储**：北斗密码保存时必须加密，密钥由服务端环境变量提供，不允许明文入库。
6. **会话缓存**：绑定或更新成功后允许缓存上游返回的 `SessionUUID` 及会话有效期信息，供后续数据查询复用。
7. **解绑能力**：用户可以解绑自己的北斗凭据；解绑后所有需要北斗/GNSS 上游数据的能力应返回可理解的授权缺失提示。
8. **授权缺失提示**：缺少用户级北斗凭据时，后续 Agent 或接口不得尝试访问北斗上游，应返回明确的绑定提示。
9. **审计与日志**：记录凭据绑定、更新、解绑、验证失败等关键动作，但日志不得记录明文密码、加密前密码或 `SessionUUID` 明文。
10. **第一阶段简化**：第一阶段不做凭据验证失败次数锁定或额外风控，依赖现有接口限流和后续安全策略扩展。

## 非目标

1. 不接入 OAuth、OIDC、LDAP、企业微信、钉钉或其他外部身份源。
2. 不实现北斗/GNSS 监测数据查询、站点解析、异常候选分析或报告生成。
3. 不实现邮件订阅、定时任务或订阅运行历史。
4. 不新增全局北斗业务账号、共享账号池或管理员代填凭据兜底机制。
5. 不向前端返回明文密码、加密密码、`SessionUUID` 或其他可直接复用的上游会话凭证。
6. 不在第一阶段实现失败次数锁定、凭据轮换提醒、密码过期主动通知或复杂风控。

## 技术方案

1. **数据模型方向**：新增用户级北斗凭据存储模型，关联本地用户，保存北斗用户名、加密后的密码、可选 `SessionUUID`、会话过期时间、创建时间和更新时间。具体字段、索引和迁移在 design 阶段明确。
2. **加密方案方向**：使用服务端环境变量提供的对称加密密钥加密北斗密码。应用启动或保存凭据时应校验密钥配置，缺少密钥时拒绝保存凭据并返回配置错误。
3. **保存前验证流程**：
   - 用户提交北斗用户名和密码。
   - 后端调用北斗 `UserLogin/doLogin.php` 登录认证接口。
   - 仅当上游返回成功码并包含有效 `SessionUUID` 时，才加密保存或更新凭据。
   - 验证失败时返回可理解错误，不保存明文或失败凭据。
4. **会话缓存方向**：绑定或更新成功后保存 `SessionUUID` 和估算有效期；后续数据查询可复用该会话。上游会话连续超过 8 小时未使用可能失效，后续查询功能设计中再处理自动重登。
5. **接口方向**：提供当前用户的凭据状态查询、绑定/更新和解绑能力。所有接口必须使用现有本地认证依赖，仅允许用户操作自己的凭据。
6. **权限边界**：缺少凭据时，北斗/GNSS 相关工具和后续 5 节点图结构中的权限门禁应识别为授权缺失，并引导用户绑定凭据。
7. **日志与审计方向**：使用 structlog，事件名遵循 lowercase_with_underscores；日志可记录本地 `user_id`、动作、北斗用户名、上游响应码和失败原因，但不得记录明文密码、加密前密码或 `SessionUUID` 明文。
8. **错误处理方向**：上游超时、网络错误、响应格式异常、认证失败、配置缺失和数据库错误应返回结构化、用户可理解的错误；预期错误使用适当 HTTP 状态码。

## 验收

- [ ] 未绑定北斗凭据的用户查询绑定状态时，返回未绑定状态。
- [ ] 已绑定北斗凭据的用户查询绑定状态时，返回已绑定状态和北斗用户名。
- [ ] 用户提交北斗用户名和密码绑定时，系统必须先调用北斗登录认证接口验证可用性。
- [ ] 北斗登录认证失败时，系统不保存或覆盖用户已有有效凭据，并返回可理解错误。
- [ ] 北斗登录认证成功时，系统加密保存北斗密码，并保存可复用的会话信息。
- [ ] 用户更新北斗凭据时，新凭据必须重新通过上游验证后才能覆盖旧凭据。
- [ ] 用户可以解绑自己的北斗凭据，解绑后状态接口返回未绑定。
- [ ] 解绑后，所有需要北斗/GNSS 上游数据的后续能力应能识别授权缺失并提示用户绑定凭据。
- [ ] 前端或 API 响应中不得返回明文密码、加密密码或 `SessionUUID`。
- [ ] 日志不得记录明文密码、加密前密码或 `SessionUUID` 明文。
- [ ] 缺少加密密钥配置时，系统拒绝保存凭据并返回配置错误。
- [ ] 本功能相关路由必须使用现有认证依赖和限流装饰器。
- [ ] 本功能相关数据库操作必须异步。
- [ ] 相关代码在后续实现阶段必须通过 `make check` 或等价的 lint/typecheck 验证。
